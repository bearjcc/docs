DriveWorks Live is delivered through three IIS websites backed by an integration theme. Front-end sites serve static files and a `config.js`; the backend site holds the integration theme and its `DriveWorksConfigUser.xml`. After any change, recheck bindings and hard-refresh to confirm.

## IIS websites

- `AutexGlobalLive` — front end
- `AutexGlobalDev` — front end
- `AutexDriveWorksMain` — backend integration theme

After changes:

1. Verify the `https` bindings and SSL certificates.
2. Verify the static file root paths.
3. Restart the website or application pool if required.
4. Validate with a browser hard refresh (<kbd>Ctrl</kbd> <kbd>F5</kbd>).

## Static file locations

| Site | Path |
| --- | --- |
| Dev | `C:\DriveWorks\StaticFiles\Autex\AutexGlobalDev` |
| Live | `C:\DriveWorks\StaticFiles\Autex\AutexGlobalLive` |
| Main | `C:\ProgramData\DriveWorks\22\Live\Themes\Integration` |

## Configuration files

- **Backend** — `DriveWorksConfigUser.xml` in the integration theme folder defines the Group and backend connection context.
- **Front end** — `config.js` in each static site folder defines the DriveApp aliases and front-end runtime settings.

## Server topology

- Administrator and Live are hosted on `AZDWLIVE01`.
- Autopilot runs on a server with SOLIDWORKS installed (`AZDWAPP01` in the current setup).
- Keeping the hosts separate reduces operational risk and clarifies troubleshooting boundaries.

## Troubleshooting

- Login page loads but content does not → verify the backend alias mapping.
- Users report stale assets → hard-refresh and confirm the deployed static file timestamps.
- Creating a new test environment → duplicate the static site folder and update both the backend and front-end config files consistently.
